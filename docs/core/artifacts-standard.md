# Artifacts Standard — docs vs data vs tools (flex-vcms)

Ten dokument jest kanonicznym standardem (PH3-006) dla artefaktów Orchestratora `flex-vcms`:
gdzie co żyje, kto jest właścicielem, czy artefakt jest **SSoT** czy **Derived**, oraz jaki ma lifecycle.

Cel: mniej śmiecenia, mniej uznaniowości, łatwiejsza weryfikacja (Gates) i szybsze decyzje operacyjne.

---

## 1) Klasyfikacja: SSoT vs Derived

### SSoT (manual, edytowane przez człowieka)

- `docs/core/*` — kontrakty, polityki, specyfikacje
- `docs/checklists/*` — gates, scenariusze testowe, checklisty
- `docs/PORADNIK_UZYTKOWNIKA.md` — SOP (entrypoint pracy)
- `docs/handoffs/*` — dowody sesji + NEXT=1 (append-only)
- `docs/journal/*` — logi sesji (append-only; format: `tmpl-session-log.md`; indeks: `docs/journal/logs-index.md`)
- `docs/ecosystem/report.md` — manualny dashboard po OP-001 (bieżący stan)
- `repos.yaml` / `scan-rules.json` — wejścia skanera (kontrakt Orchestratora)

### Derived (generated, nadpisywane przez narzędzie)

Generated przez OP-001 (`node tools/vcms-scan.js`):

- `data/vcms-index.json`
- `docs/ecosystem/conflicts.md`
- `docs/ecosystem/map.md`
- `docs/ecosystem/repos/*.md`

---

## 2) Ownership (kto odpowiada)

- **Owner: Orchestrator (`flex-vcms`)**
  Wszystko w `docs/ecosystem/*` oraz `data/vcms-index.json` (snapshot stanu, derived artefakty skanu).

- **Owner: Dowódca**
  Decyzje „co jest canonical” (źródła prawdy w ekosystemie) i priorytety pracy:
  `repos.yaml`, `brain.md`, backlog (`flex-vcms-todo.json`).

---

## 3) Lifecycle (nadpisywanie vs wersjonowanie)

### Derived (scan outputs)

- **Nadpisywane** przy każdym OP-001.
- Traktuj jako **aktualny snapshot** (nie historia).
- Historia „co i dlaczego” ma trafiać do **handoffów**, nie do mapy/conflicts.

### Manual (SSoT)

- `docs/handoffs/*` — **append-only** (nie edytujemy wstecz bez jawnego powodu).
- `docs/ecosystem/report.md` — **nadpisywane/aktualizowane** jako bieżący dashboard.
- `docs/core/*` / `docs/checklists/*` — edytowane w PR flow (review wymagany).

---

## 4) Zasady commitowania (żeby repo było używalne)

### Commitujemy

- Wszystkie **SSoT** dokumenty (core/checklists/SOP/handoffs/report).
- Derived artefakty skanu jako lokalny dashboard repo:
  - `docs/ecosystem/*`
  - `data/vcms-index.json`
  pod warunkiem zgodności z `scan-rules.json` (brak sekretów, brak binarek).

### Nie commitujemy (P0)

- Czegokolwiek z denylist (sekrety, binarki, archiwa), w szczególności:
  `.env*`, `wp-config.php`, klucze/certyfikaty, archiwa, binarki.
- Cache/tymczasowe śmieci: `node_modules`, `dist/build`, `.git`, `archive/*`, itp.

Źródło: `scan-rules.json` + `docs/core/security-policy.md`.

---

## 5) Naming conventions (minimalne)

- **Handoffs**: `docs/handoffs/YYYY-MM-DD-<topic>.md`
- **Repo pages**: `docs/ecosystem/repos/<slug>.md` (slug deterministyczny, bez spacji)
- **Ecosystem report**: `docs/ecosystem/report.md` (jeden plik = single dashboard)
